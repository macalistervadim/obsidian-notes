### Определение

> **Принцип наименьших привилегий** означает, что пользователь, сервис, процесс или компонент системы должен иметь только те права, которые нужны ему для выполнения текущей задачи, и не больше.

Идея простая:

- если компонент взломают, ущерб будет ограничен
- если разработчик ошибётся, последствия будут меньше
- если доступ утечёт, злоумышленник не получит полный контроль над системой

---
### Зачем это нужно в вебе

Веб-приложение обычно состоит не из одного сервера, а из набора частей:

- frontend
- backend
- база данных
- кеш
- очереди
- сторонние API
- админка
- CI/CD
- облачная инфраструктура

Если где-то дать избыточные права, одна уязвимость может превратиться в полный захват системы.

Пример:

- обычный API-сервис должен читать и изменять только свои таблицы
- но если ему выдать `SUPERUSER` в БД, то при взломе атакующий сможет читать всё, удалять данные и менять схему базы

---
### Основная логика принципа

Нужно отвечать на вопрос:

> Какие минимальные права реально нужны этому субъекту для выполнения конкретного действия?

Субъектом может быть:

- пользователь
- администратор
- сервисный аккаунт
- контейнер
- микросервис
- cron-задача
- CI job
- API token

---
### Где применять в веб-приложении

#### 1. Права пользователей

Не всем пользователям нужен одинаковый доступ.

Пример ролей:

- гость: только просмотр публичных страниц
- пользователь: работа только со своими данными
- модератор: ограниченная модерация контента
- администратор: управление системой

Правильный подход:

- пользователь может редактировать только свой профиль
- не может читать чужие заказы, сообщения, документы

Это напрямую снижает риск `IDOR` и горизонтального повышения привилегий.

---
#### 2. Права внутри backend

Разные части системы не должны иметь одинаково широкий доступ.

Примеры:

- сервис загрузки файлов не должен иметь доступ к платежам
- сервис отправки email не должен иметь доступ на удаление пользователей
- воркер отчётов не должен иметь админские права в основном API

---
#### 3. Права к базе данных

Частая ошибка:

- приложение подключается к БД под пользователем с полными правами

Лучше:

- `SELECT` только там, где нужно только чтение
- `INSERT/UPDATE` только для нужных таблиц
- отдельный пользователь для миграций
- отдельный readonly-доступ для аналитики

Идея:

- runtime-приложение не должно менять схему БД
- миграции не должны выполняться от имени обычного приложения

---
#### 4. Права к API и токенам

API-ключи и токены должны быть узкими по области действия.

Правильно:

- отдельные токены под конкретный сервис
- ограничение по scope
- ограничение по времени жизни
- ротация токенов

Плохо:

- один вечный токен с полным доступом ко всем данным и окружениям

---
#### 5. Права в инфраструктуре

Примеры:

- контейнер не должен запускаться от `root`, если это не требуется
- pod в Kubernetes не должен получать лишние capabilities
- приложение не должно иметь доступ ко всем секретам namespace
- CI не должен иметь право деплоить во все окружения сразу

---
### Пример из практики

Плохой вариант:

- Django/Flask/FastAPI приложение работает с БД как `postgres`
- серверный процесс запущен от `root`
- `.env` содержит ключи от продакшна, staging и S3 одновременно
- админка доступна слишком широкому кругу сотрудников

Что будет при компрометации:

- атакующий получит максимум возможностей почти без дополнительных шагов

Хороший вариант:

- отдельный пользователь БД только для приложения
- отдельный пользователь БД для миграций
- process user без root-прав
- секреты разделены по окружениям
- роли пользователей ограничены бизнес-логикой
- действия админа логируются

---
### Что даёт принцип наименьших привилегий

- уменьшает поверхность атаки
- ограничивает масштаб ущерба
- мешает вертикальному повышению привилегий
- упрощает аудит доступов
- делает систему предсказуемее и безопаснее

---
### Типичные ошибки

- у всех сервисов одинаковые учётки
- приложение работает от root/admin
- одна роль "admin" используется слишком часто
- отсутствие разделения прав на чтение и изменение
- доступ "на всякий случай"
- старые токены и права не отзываются
- dev/stage/prod не разделены по доступам

---
### Мини-чеклист

- у каждого сервиса отдельная учётка
- у БД-пользователей минимально нужные права
- роли пользователей разделены по действиям
- доступ к чужим объектам всегда проверяется на уровне backend
- секреты и токены ограничены по scope
- root/admin доступ выдаётся редко и осознанно
- лишние права регулярно пересматриваются

---
### Краткое резюме

>[!info] Принцип наименьших привилегий в веб-безопасности означает: каждому пользователю и компоненту выдаются только минимально необходимые права, чтобы при ошибке или взломе атакующий не получил лишний доступ.
