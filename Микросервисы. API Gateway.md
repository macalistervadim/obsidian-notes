  ## Суть  
  
**API Gateway** — это единая точка входа во все микросервисы системы.  
  
Он принимает все внешние запросы и:  
- маршрутизирует их в нужные сервисы  
- выполняет инфраструктурные задачи (auth, логирование и т.д.)  
  
---  
  
## Зачем нужен API Gateway  
  
Без gateway:  
  

Frontend → users-service  
Frontend → portfolio-service  
Frontend → analytics-service  
Frontend → notifications-service

Проблемы:

- фронт знает внутреннюю архитектуру
- много точек входа
- сложнее контролировать безопасность
- сложнее менять бэк

---

С gateway:

Frontend → API Gateway → микросервисы

Плюсы:

- единая точка входа
- скрытие внутренней архитектуры
- централизованный контроль
- гибкость при изменениях

---

## Основные функции

### 1. Роутинг (Routing)

Определяет, куда отправить запрос:

/api/users/*      → users-service  
/api/portfolio/*  → portfolio-service  
/api/analytics/*  → analytics-service

---

### 2. Аутентификация и авторизация

- проверка JWT / токенов
- извлечение user_id
- прокидывание данных в сервисы

👉 сервисы становятся проще

---

### 3. Rate Limiting

- ограничение количества запросов
- защита от перегрузки и атак

---

### 4. Логирование и мониторинг

- все запросы проходят через одну точку
- удобно собирать метрики:
    - latency
    - ошибки
    - нагрузку

---

### 5. TLS / HTTPS

- управление сертификатами
- шифрование соединений

---

### 6. Версионирование API

/api/v1/...  
/api/v2/...

---

### 7. Преобразование запросов (опционально)

- добавление заголовков
- изменение путей
- прокидывание контекста

---

## Чего API Gateway НЕ делает

❌ не реализует бизнес-логику  
❌ не агрегирует данные из сервисов  
❌ не знает доменную модель  
❌ не заменяет backend

Если это происходит → он превращается в “god object”

---

## Разница: API Gateway vs BFF

### API Gateway

- инфраструктурный слой
- маршрутизация
- безопасность
- “тупой, но быстрый”

---

### BFF (Backend for Frontend)

- слой под конкретный UI
- агрегирует данные
- формирует удобные ответы

---

### Вместе:

Frontend → API Gateway → BFF → микросервисы

---

## Пример для eebook

### Сейчас (антипаттерн при росте)

Frontend:  
  → users-service  
  → portfolio-service  
  → analytics-service  
  → notifications-service

---

### С gateway

Frontend → API Gateway → сервисы

---

### С gateway + BFF (рекомендуется)

Frontend → API Gateway → BFF → микросервисы

BFF:

GET /dashboard  
  → собирает данные из 5 сервисов  
  → возвращает один ответ

    proxy_pass http://users-service/api/v1/;

location /api/portfolio/ {  
    proxy_pass http://portfolio-service/api/v1/;  
}

    proxy_pass http://analytics-service/api/v1/;  
}

## Надёжность и отказоустойчивость

❗ Gateway — это не одна точка отказа, если сделан правильно

### Правильная схема:

Frontend → Load Balancer → [Gateway x N] → сервисы

- несколько инстансов gateway
- health checks
- автоматическое исключение упавших нод

---

## Когда достаточно nginx

Подходит для:

- простого routing
- reverse proxy
- базового gateway

---

## Когда нужен “продвинутый” gateway

- сложная авторизация
- rate limiting
- circuit breaker
- observability

Тогда используют:

- Kong
- Traefik
- Envoy
- AWS API Gateway

---

## Лучшие практики

- не светить наружу названия сервисов (`users-service`)
- использовать доменные пути (`/api/users`)
- не добавлять бизнес-логику в gateway
- держать его stateless
- масштабировать горизонтально

---

## Антипаттерны

❌ фронт знает все сервисы  
❌ gateway содержит бизнес-логику  
❌ один инстанс gateway в проде  
❌ длинные цепочки синхронных вызовов  
❌ отсутствие централизованного входа

---

## Выжимка

- API Gateway = вход в систему
- решает инфраструктурные задачи
- скрывает микросервисы от клиента
- НЕ занимается агрегацией

👉 для сложных экранов нужен BFF
