> Планировщик k8

Планировщик нужен для того, чтобы нужному поду выделять нужное кол-во ресурсов, необходимых для его жизнедеятельности. Мы можем строго задавать в манифестах, кому и сколько необходимо ресурсов - это очень рекомендуется в продакшн-среде, ведь если мы не выделяем определенное кол-во ресурсов, то распределение выглядит примерно так как на рисунке.

![[Kubernetes для разработчиков 4.jpeg]]

Тоесть высок риск того, что поду может попросту не хватить оставшихся ресурсов.

Помимо выделенных - также есть и лимиты по ресурсам. Это указатели *максимально допустимых значений*, которые планировщик может выделить тому или иному поду. Но cpu и memory ведут себя по разному при достижении этих лимитов.

Рассмотрим на примере:
```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: k8-test  
spec:  
  replicas: 3  
  selector:  
    matchLabels:  
      pod: k8-test-pod  
  template:  
    metadata:  
      labels:  
        pod: k8-test-pod  
    spec:  
      containers:  
        - name: k8-test-container  
          image: docker.io/macalistervadim/k8-test:latest  
          resources:  # вот здесь мы и задаем ресурсы
            requests:  # которые нам необходимы
              cpu: 200m  
              memory: 250Mi  
            limits:  # и наши лимиты
              cpu: 300m  
              memory: 400Mi
```

Короткие заметки про requests и limits (простыми словами)

- CPU и память — это «резурсы» узла. Подам их нужно делить.
- requests — минимально нужный запас:
    - Говорим планировщику: “мне нужно не меньше вот столько”.
    - По этим цифрам решают, поместится ли под на узел.
    - Это не «жёсткий выкуп»: если вы тратите меньше, свободное могут использовать другие.
- limits — потолок:
    - Больше этого потреблять нельзя.
    - Для CPU: выше лимита начнётся «торможение» (throttling).
    - Для памяти: при попытке выйти за лимит контейнер упадёт (OOMKill).
- Что если тратим меньше, чем requests?
    - В моменте узел отдаст свободное другим подам.
    - Но новые поды могут не пустить на этот узел, потому что планировщик считает ваш request занятым.
- Что если хотим больше, чем limits?
    - CPU: не дадут — будете ограничены лимитом (медленнее работать).
    - Память: попытка взять больше — падение контейнера (перезапуск).
- Классы качества обслуживания (QoS):
    - Guaranteed: requests = limits для CPU и RAM. Выселяют последним.
    - Burstable: есть requests и limits, но они не равны. *Наш случай*.
    - BestEffort: ничего не задано. Выселяют первым.
- Практика настройки:
    - Requests ставьте под стабильную среднюю нагрузку.
    - Limits — под безопасный пик (обычно 1.5–3× от requests).
    - Критичное — делайте Guaranteed (requests == limits), чтобы меньше рисков.
    - Часто CPU-лимит лучше не ставить (оставить только request), чтобы избежать искусственного торможения; но память-лимит ставьте обязательно, чтобы не «съесть» весь узел.
    - Мониторьте реальные метрики и корректируйте.

Пример наших значений:

- requests: CPU 200m, RAM 250Mi — планировщик резервирует под это место.
- limits: CPU 300m, RAM 400Mi — выше этого не подниметесь (CPU — будет тормозить, RAM — упадёт).
