# Переход от монолита к микросервисам. Паттерн «Душитель» (Strangler Fig)  
  
## Суть паттерна  
  
**Strangler Fig (паттерн «Душитель»)** — это способ постепенной миграции от монолита к микросервисам без переписывания системы с нуля.  
  
Идея:  
> Новая функциональность выносится в отдельные сервисы, которые постепенно "обрастают" вокруг монолита, пока он не будет полностью заменён.  
  
---  
  
## Ключевой принцип  
  
❗ **Монолит не переписывается сразу — он "умирает" постепенно**  
  
- новые фичи → сразу в микросервисы    
- старые части → постепенно вырезаются и заменяются    
- монолит → со временем становится всё меньше    
  
---  
  
## Общая схема перехода  
  
1. Перед монолитом ставится **единая точка входа**  
   - API Gateway / BFF / Reverse Proxy  
  
2. Все запросы проходят через неё  
  
3. Далее:  
   - если функциональность уже вынесена → запрос уходит в микросервис  
   - если нет → в монолит  

Клиент → Gateway → [Микросервис ИЛИ Монолит]

  
---  
  
## Пошаговый процесс  
  
### 1. Введение прослойки (Gateway)  
  
- Nginx / API Gateway / Backend-for-Frontend  
- начинает маршрутизировать трафик  
  
---  
  
### 2. Выделение первого сервиса  
  
Берётся **наименее связанный кусок**:  
- авторизация  
- уведомления  
- платежи  
- профиль пользователя  
  
👉 Выносится в отдельный сервис  
  
---  
  
### 3. Перенаправление трафика  
  
- раньше: `GET /users → монолит`  
- теперь: `GET /users → сервис пользователей`  
  
---  
  
### 4. Декомпозиция по доменам  
  
Постепенно выносится:  
- orders  
- billing  
- catalog  
- analytics  
  
Каждый кусок:  
- получает свой API  
- отвязывается от монолита  
  
---  
  
### 5. Удаление кода из монолита  
  
После миграции:  
- старый код удаляется  
- зависимости вычищаются  
  
---  
  
### 6. Финальный этап  
  
Монолит:  
- либо полностью исчезает  
- либо остаётся как "остаточный сервис"  
  
---  
  
## Важные техники внутри паттерна  
  
### 1. Routing / Proxy  
  
- определяет куда идёт запрос  
- ключевая точка контроля миграции  
  
---  
  
### 2. Anti-Corruption Layer (ACL)  
  
Прослойка между монолитом и микросервисами:  
- защищает новый код от "грязной" модели монолита  
- маппит DTO / модели  
  
---  
  
### 3. Database decomposition  
  
❗ Самая сложная часть  
  
Варианты:  
- сначала общий доступ к БД (временно)  
- затем постепенное разделение данных  
  
---  
  
### 4. Data synchronization  
  
Когда часть данных уже в сервисе:  
- CDC (Change Data Capture)  
- события (event-driven)  
- репликация  
  
---  
  
## Стратегии выноса  
  
### 🔹 По функциональности (рекомендуется)  
- Users  
- Orders  
- Payments  
  
### 🔹 По слоям (плохо)  
- не делить на "API сервис" и "DB сервис"  
  
---  
  
## Плюсы паттерна  
  
- нет "big bang" переписывания  
- можно релизить постепенно  
- контролируемые риски  
- бизнес не останавливается  
  
---  
  
## Минусы и риски  
  
- временная сложность ↑  
- дублирование логики  
- консистентность данных усложняется  
- нужна грамотная маршрутизация  
  
---  
  
## Когда применять  
  
✔ большой монолит    
✔ система в проде    
✔ нельзя остановить бизнес    
✔ нужна постепенная миграция    
  
❌ если проект маленький → проще переписать    
  
---  
  
## Коротко (выжимка)  
  
- ставим Gateway    
- начинаем выносить куски    
- маршрутизируем трафик    
- удаляем старый код    
- повторяем    
  
👉 пока монолит не исчезнет
