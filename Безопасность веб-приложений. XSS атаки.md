### Определение

> **XSS (Cross-Site Scripting)** — это тип атаки, при котором злоумышленник внедряет вредоносный **JavaScript-код** в веб-страницу, которая затем выполняется в браузере другого пользователя.

❗ Важно:

- XSS — это **не про сервер напрямую**, а про **браузер жертвы**
- Код выполняется **от имени доверенного сайта**

---
###  Суть уязвимости

XSS возникает, когда приложение:

- принимает пользовательский ввод
- **не очищает (sanitize) и не экранирует (escape) его**
- и затем **вставляет в HTML-страницу**

В результате браузер воспринимает этот ввод как код, а не как данные.

---

### Простой пример

`<input value="{{ user_input }}">`

Если пользователь введёт:

<script>alert('XSS')</script>

и сервер это вставит без обработки — скрипт выполнится.

---

### Основные типы XSS

#### 1. Stored XSS (сохранённая)

Самая опасная.

📌 Сценарий:

- злоумышленник сохраняет вредоносный код (например, в БД)
- другие пользователи открывают страницу
- скрипт выполняется у них

Пример:

- комментарии
- профили пользователей

---

#### 2. Reflected XSS (отражённая)

📌 Сценарий:

- вредоносный код передаётся через URL
- сервер отражает его в ответе

 Пример:

https://site.com/search?q=<script>alert(1)</script>

---

#### 3. DOM-based XSS

📌 Сценарий:

- уязвимость на стороне клиента (JS)
- сервер тут вообще не участвует

Пример:

document.innerHTML = location.hash

---
### Что может сделать атакующий

- украсть **cookies** (если нет HttpOnly)
- украсть **session**
- выполнять действия от имени пользователя
- делать фишинг внутри сайта
- подменять UI
- редиректить пользователя
- внедрять кейлоггеры

---

### Защита от XSS

#### 1. Экранирование (Output Escaping)

Самое важное правило:

> ❗ Никогда не вставляй пользовательский ввод в HTML без экранирования

Пример:
```
&lt;script&gt; вместо <script>
```
---

#### 2. Валидация входных данных

- whitelist лучше blacklist
- проверка типов и форматов

---

#### 3. Content Security Policy (CSP)

HTTP заголовок:

Content-Security-Policy: script-src 'self'

запрещает выполнение сторонних скриптов

---

#### 4. HttpOnly cookies

Set-Cookie: session=abc; HttpOnly

JS не сможет их прочитать

---

#### 5. Использование безопасных фреймворков

- React / Vue автоматически экранируют данные
- но опасные места всё равно есть (`dangerouslySetInnerHTML`)

---

###  Частые ошибки разработчиков

- доверие данным из формы
- использование `innerHTML`
- отсутствие CSP
- ручная сборка HTML строк

---

### Краткое резюме

>[!info] XSS — это выполнение вредоносного JS-кода в браузере пользователя из-за неочищенного пользовательского ввода.
