> Общая связанность возникает в ситуации, когда несколько модулей или микросервисов используют **одну и ту же общую область данных**, которая может изменяться любым из них.

В такой архитектуре сервисы **зависят от одного общего состояния**, что приводит к сильной связанности между компонентами системы.

Главная проблема заключается в том, что **изменения в данных или их структуре могут затронуть сразу несколько сервисов**, даже если эти сервисы логически не связаны друг с другом.

---

# Основная идея

Общая связанность возникает когда:

Service A  
Service B  
Service C  
      ↓  
 Shared Data

Каждый сервис:

- может читать данные
- может изменять данные
- зависит от структуры этих данных

В результате сервисы **не являются изолированными**.

---

# Основные проблемы

### 1. Нарушение изоляции сервисов

В микросервисной архитектуре каждый сервис должен владеть своими данными:

service owns its data

При общей связанности это правило нарушается, поскольку несколько сервисов используют одно и то же хранилище данных.

---

### 2. Каскадные изменения

Если меняется структура данных:

shared_table

например:

users.email → users.primary_email

то изменения могут затронуть:

Service A  
Service B  
Service C

Даже если часть сервисов использует эти данные косвенно.

---

### 3. Потеря контроля над данными

Когда несколько сервисов имеют доступ на запись:

Service A → write  
Service B → write  
Service C → write

становится сложно определить:

- кто изменил данные
- какие правила должны применяться
- где должна находиться бизнес-логика

Это может привести к нарушению **доменных инвариантов**.

---

### 4. Рост связанности системы

Чем больше сервисов используют общие данные, тем сложнее становится:

- изменять архитектуру
- масштабировать систему
- разделять сервисы

---

# Пример общей связанности

Неправильная архитектура:

Service A  
Service B  
Service C  
      ↓  
  Shared Database

Все сервисы работают с одними и теми же таблицами.

Например:

orders  
users  
payments

Это фактически превращает микросервисы в **распределенный монолит**.

---

# Правильная архитектура

В микросервисах каждый сервис должен иметь **собственное хранилище данных**:

Service A → Database A  
Service B → Database B  
Service C → Database C

Взаимодействие происходит через:

- REST API
- gRPC
- события (event bus)

---

# Общая инфраструктура vs общие данные

Важно различать:

shared infrastructure

и

shared data

Например:

| Компонент             | Нарушение? |
| --------------------- | ---------- |
| Shared Redis cache    | нет        |
| Shared message broker | нет        |
| Shared logging system | нет        |
| Shared database       | да         |

Общая инфраструктура допустима, пока сервисы **не делят одно и то же изменяемое состояние**.

---

# Как избежать общей связанности

### 1. Владение данными

Каждый сервис должен владеть собственными данными:

portfolio-service → portfolio DB  
assets-service → assets DB

---

### 2. Взаимодействие через API

Доступ к данным другого сервиса должен происходить только через его интерфейс:

portfolio-service → assets-service API

---

### 3. Использование событий

Иногда лучше передавать изменения через события:

order_created  
portfolio_updated  
price_changed

---

# Кратко

Общая связанность возникает, когда:

несколько сервисов используют одно и то же изменяемое состояние

Основные проблемы:

- нарушение изоляции сервисов
- каскадные изменения
- потеря контроля над данными
- рост связанности системы

Правильный подход:

each service owns its data

и взаимодействует с другими сервисами только через **контракты (API или события)**.
