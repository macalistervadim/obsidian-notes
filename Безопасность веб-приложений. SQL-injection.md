### Определение

> **SQL Injection (SQLi)** — это тип атаки, при котором злоумышленник внедряет вредоносный SQL-код в запрос к базе данных через пользовательский ввод.

---
###  Суть уязвимости

SQL-инъекция возникает, когда:

- приложение принимает пользовательский ввод
- **вставляет его в SQL-запрос без параметризации**
- и выполняет этот запрос

👉 В результате пользователь может:

- изменить структуру запроса
- получить доступ к данным
- изменить или удалить данные

---

### Простой пример

query = f"SELECT * FROM users WHERE login = '{login}' AND password = '{password}'"

Если пользователь введёт:

login: admin  
password: ' OR 1=1 --

Запрос станет:

SELECT * FROM users WHERE login = 'admin' AND password = '' OR 1=1 --'

👉 `OR 1=1` всегда true → вход без пароля

---

### Основные типы SQL-инъекций

#### 1. In-band SQLi (классическая)

Данные возвращаются в ответе.

- **Union-based**

' UNION SELECT username, password FROM users --

- **Error-based**
    - получение данных через ошибки БД

---

#### 2. Blind SQL Injection

Когда нет прямого ответа от БД.

- **Boolean-based**

' AND 1=1 --  
' AND 1=2 --

→ по разнице в ответах угадываются данные

- **Time-based**

' AND SLEEP(5) --

→ если сервер "завис", значит условие истинно

---

#### 3. Out-of-band SQLi

Редкий случай:

- данные отправляются через внешние каналы (DNS, HTTP)

---

### Что может сделать атакующий

- получить доступ к базе данных
- прочитать таблицы (логины, пароли, токены)
- изменить данные
- удалить данные
- обойти аутентификацию
- в некоторых случаях — выполнить команды на сервере

---

### Типичные причины

- использование строковой конкатенации в SQL
- отсутствие параметризованных запросов
- доверие пользовательскому вводу
- слабая архитектура доступа к БД

---

###  Защита от SQL-инъекций

#### 1. Параметризованные запросы (самое важное)

cursor.execute(  
    "SELECT * FROM users WHERE login = %s AND password = %s",  
    (login, password)  
)

👉 данные и SQL-код разделены

---

#### 2. ORM (но с пониманием)

- SQLAlchemy, Django ORM и т.д.
- по умолчанию защищают

 но:

- raw SQL = снова риск

---

#### 3. Валидация входных данных

- типы (int, email и т.д.)
- длина
- whitelist

---

#### 4. Ограничение прав БД

- не использовать root
- принцип минимальных привилегий

---

#### 5. Логирование и мониторинг

- подозрительные запросы
- частые ошибки SQL

---

### Частые ошибки разработчиков

- f-строки / `.format()` в SQL
- "я проверил input, значит безопасно"
- использование ORM + raw SQL без параметров
- отсутствие ограничения прав БД

---

### Краткое резюме

> SQL-инъекция — это возможность изменить SQL-запрос через пользовательский ввод из-за отсутствия параметризации.
