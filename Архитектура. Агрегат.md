> **Агрегат** — это группа связанных доменных объектов (сущностей и value objects), объединённых вокруг одного главного объекта — **корня агрегата (Aggregate Root)**.

Агрегат задаёт **границу согласованности (consistency boundary)**:  
все изменения внутри агрегата происходят как единое целое, гарантируя целостность и непротиворечивость данных.

Таким образом, агрегат — это **атомарная единица изменения в доменной модели**.

---

#  Зачем нужен агрегат

Агрегат существует не просто для группировки объектов, а для того, чтобы:

- определить **границу бизнес-инвариантов**
- ограничить область **сильной согласованности**
- контролировать изменения состояния
- упростить управление транзакциями

> Агрегат — это минимальная часть модели, которая должна всегда оставаться в валидном состоянии.

---

# Ключевые принципы

### 1. Корень агрегата

- В агрегате всегда есть **Aggregate Root**
- Внешний код взаимодействует **только с ним**
- Он контролирует все изменения

external code → Aggregate Root → internal objects

---

### 2. Инварианты

Внутри агрегата соблюдаются **бизнес-правила (инварианты)**, которые нельзя нарушать.

Примеры:

- заказ нельзя подтвердить без позиций
- количество не может быть отрицательным
- нельзя изменять завершённую сущность

---

### 3. Изменения только через корень

Внутренние объекты агрегата:

- не должны изменяться напрямую извне
- изменяются только через методы корня агрегата

Это гарантирует, что инварианты не будут нарушены.

---

### 4. Атомарность изменений

Все изменения внутри агрегата:

- выполняются как единое целое
- завершаются в **валидном состоянии**

> После любой операции агрегат должен оставаться консистентным.

---

### 5. Граница транзакции

Агрегат — это:

transaction boundary

- изменения внутри агрегата — в одной транзакции
- между агрегатами — не обязательно

---

### 6. Межагрегатная согласованность

Между разными агрегатами:

- часто используется **eventual consistency**
- взаимодействие через:
    - события
    - сервисы
    - API

---

### 7. Ссылки между агрегатами

Агрегаты **не должны включать друг друга напрямую**.

Правильно:

order.customer_id

Неправильно:

order.customer  # объект другого агрегата

Это снижает связанность и упрощает систему.

---

### 8. Размер агрегата

Агрегат должен быть:

> настолько маленьким, насколько это возможно,  
> но настолько большим, насколько это необходимо для сохранения инвариантов.

Слишком большой агрегат → проблемы:

- блокировки
- сложные транзакции
- плохая масштабируемость

---

### 9. Репозиторий

Репозиторий работает с агрегатом как с целым:

OrderRepository → работает с Order (а не с OrderLine отдельно)

---

# Пример

```python
class OrderLine:  
    def __init__(self, sku: str, qty: int):  
        self.sku = sku  
        self.qty = qty  
  
  
class Order:  
    def __init__(self, order_id: str):  
        self.order_id = order_id  
        self.lines: list[OrderLine] = []  
        self.status = "draft"  
  
    def add_line(self, sku: str, qty: int) -> None:  
        if self.status != "draft":  
            raise ValueError("Cannot modify a confirmed order")  
  
        if qty <= 0:  
            raise ValueError("Quantity must be positive")  
  
        self.lines.append(OrderLine(sku, qty))  
  
    def confirm(self) -> None:  
        if not self.lines:  
            raise ValueError("Order cannot be confirmed without lines")  
  
        self.status = "confirmed"  
  
    def total_quantity(self) -> int:  
        return sum(line.qty for line in self.lines)
```
---

#  Что здесь важно

- **Order** — корень агрегата
- `OrderLine` не изменяется напрямую извне
- все изменения проходят через `Order`
- инварианты:
    - нельзя добавить qty ≤ 0
    - нельзя менять подтверждённый заказ
    - нельзя подтвердить пустой заказ

---

# Антипаттерны

### ❌ Прямой доступ к внутренним объектам

order.lines.append(...)

→ обход инвариантов

---

### ❌ Огромные агрегаты

Order + Customer + Payment + Shipment

→ превращается в монолит внутри сервиса

---

### ❌ Shared агрегаты между сервисами

несколько сервисов работают с одной моделью

→ нарушение границ

---

# Коротко

Агрегат — это:

- граница согласованности
- защитник инвариантов
- атомарная единица изменений
- точка управления состоянием

---

# Главное правило

> Если правило должно соблюдаться всегда — оно должно находиться внутри одного агрегата.
