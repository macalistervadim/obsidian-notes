> Сквозная связанность возникает в ситуации, когда один микросервис передает данные другому микросервису не потому, что эти данные нужны ему самому, а исключительно потому, что они требуются третьему сервису, находящемуся дальше по цепочке взаимодействия.

В такой архитектуре первый сервис фактически **становится посредником**, который транзитом передает данные между другими сервисами.

Главная проблема заключается в том, что сервис начинает зависеть **от внутренних требований другого сервиса**, с которым он напрямую даже не взаимодействует.

Это приводит к нарушению принципа **слабой связанности (loose coupling)** между микросервисами.

---

# Основные проблемы

### 1. Расширение контрактной зависимости

Посредник вынужден знать структуру данных, которые фактически предназначены для другого сервиса.

В результате:

- меняется контракт конечного сервиса
- необходимо изменить и посредника

Это увеличивает **связанность между сервисами**.

---

### 2. Каскадные изменения

Если меняются требования конечного сервиса:

Service A → Service B → Service C

например Service C начинает требовать дополнительные поля,

то потребуется изменить и развернуть:

Service B  
Service A

Хотя **Service A фактически не использует эти данные**.

---

### 3. Расширение области ответственности

Посредник начинает выполнять не свою доменную задачу, а становится **транспортным слоем для чужих данных**.

Это нарушает принцип:

service owns its data and contract

---

# Пример сквозной связанности

API → Order Service → Payment Service

Если `Payment Service` требует поле:

payment_method

то `Order Service` вынужден:

- принять это поле
- передать его дальше

Хотя оно не относится к домену заказов.

---

# Способы уменьшения сквозной связанности

### 1. Прямое взаимодействие сервисов

Если архитектура позволяет, лучше избегать посредников:

Service A → Service C

вместо

Service A → Service B → Service C

Это уменьшает транзитные зависимости.

---

### 2. Разделение ответственности API

Посредник должен принимать **только те данные, которые относятся к его домену**, а другие сервисы должны запрашивать необходимые данные самостоятельно.

Например:

API → Order Service  
API → Payment Service

или

Payment Service → Order Service

---

### 3. Event-driven взаимодействие

Вместо передачи данных по цепочке можно публиковать событие:

order_created

и другие сервисы самостоятельно реагируют на него.

Order Service → event bus → Payment Service

Это уменьшает синхронную связанность.

---

### 4. BFF или API Gateway

Иногда правильным решением является вынесение оркестрации на уровень gateway:

client  
  ↓  
API Gateway  
  ├ Order Service  
  └ Payment Service

Тогда сервисы **не передают чужие данные друг через друга**.
